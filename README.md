# 律疏 · 中国法律助手

中国法律法规检索与智能咨询 iOS 应用，数据来源于 [laws-data](https://github.com/13098806890/laws-data) 开放数据集。

**当前版本：1.05** — [Release Notes](CHANGELOG.md)

---

## 功能

### 法律浏览
- 按法律部门（民法、刑法、行政法等）分类浏览约 **1,572 部**现行有效法规（含高法司法解释）
- 编 / 章 / 节 / 条 层级导航，支持目录跳转与条文锚点滚动
- 条文内容全文展示，支持字号调节（小 / 中 / 大 / 超大）
- 条文收藏（书签）与 iCloud 多设备同步
- 法条下方显示关联公报案例链接，点击跳转对应文书

### 全文搜索
- FTS5 trigram 索引，支持任意中文短语搜索（≥3 字）
- FTS5 unicode61 索引，支持 1–2 字短词搜索
- 结果按法律部门过滤，关键词高亮显示
- 支持仅搜标题 / 含条文内容两种模式

### 人民法院公报
- 收录最高人民法院公报全量数据，共 2,289 篇
  - **指导案例**（`al`）：986 篇，含裁判要点摘要和关键词
  - **司法文件**（`sfwj`）：860 篇
  - **裁判文书**（`cpwsxd`）：443 篇
- 关键词全文搜索（FTS5 trigram）
- 案例笔记：为文书添加个人标注，辅助 AI 咨询时的相关案例推荐
- 收藏案例，iCloud 多设备同步

### 法律顾问（AI）
- 意图识别自动路由四种模式（详见下方"Agent 路由规则"）
- **多专家协作**：协调员 → 6 个专家组 → 17 个细分领域专家，追问补全关键信息（最多 1–5 轮，可在设置中配置）
- **关联公报案例**：回答结束后自动检索相关公报案例，以卡片形式展示，点击查看完整文书
- 对话历史持久化，支持跨 session 恢复
- 对话内容导出为文本或 PDF（含参考法条与公报案例）

## Agent 路由规则

### 第一步：意图分类（单次 LLM 调用）

每条用户消息先经过意图分类，结合当前对话历史，输出以下四类之一：

| 意图 | 触发条件 | 后续处理 |
|------|---------|---------|
| `off_topic` | 与法律无关的问题 | 直接拒绝，不消耗次数 |
| `follow_up` | 历史对话中已有明确案情，当前是追问 | 跳过专家选择，直接用上轮专家组继续分析 |
| `legal_query` | 法律相关问题 | 进入模式分类 → 专家路由 |

`follow_up` 的判定有严格前提：**历史对话中必须存在明确具体的案情事实**，否则归入 `legal_query`。

---

### 第二步：模式分类（与意图分类合并为同一次 LLM 调用）

`legal_query` 类问题同时输出 `mode`，决定后续走哪条处理路径：

| 模式 | 枚举值 | 触发关键词 / 条件 | 处理路径 |
|------|--------|-----------------|---------|
| 案情分析 | `case` | 描述了具体当事人、事件经过、时间节点 | 追问 → 多专家并行分析 → 综合协调 |
| 法律咨询 | `advisory` | 假设场景的权利义务询问，无具体案情 | 不追问，直接检索 → 多专家分析 |
| 法条检索 | `conceptAndStatute` | 查询法律规定本身，无案情也无明确指向公报 | 不追问，RAG 检索 → 协调员综合 |
| 公报检索 | `gongbao` | 问句含"案例"、"判决"、"指导案例"、"裁判文书"、"司法文件"、"法院怎么判"、"实践中"、"有没有案例"等 | 跳过专家，直接公报检索 → 结果汇总 |

---

### 第三步：专家路由（案情分析 / 法律咨询 / 法条检索路径）

#### 专家组分配

LLM 根据问题内容，从 6 个专家组中选取 1–3 个相关组，每组包含若干细分专家：

| 专家组 | 细分专家 | 路由关键词（示例） |
|--------|---------|----------------|
| 民法专家组 | 合同通则、买卖合同、租赁合同、借款合同、建设工程合同、服务委托合同、离婚财产分割、法定继承、名誉隐私权、不动产所有权、侵权通则、医疗损害责任、产品责任、交通事故责任、特殊侵权 | 合同、违约、租赁、侵权、赔偿、离婚、继承、名誉、物权 |
| 刑法专家组 | 财产犯罪 | 犯罪、刑事、诈骗、盗窃、故意伤害、量刑、构成要件 |
| 劳动法专家组 | 劳动合同 | 劳动、工资、辞退、工伤、经济补偿、试用期、社保 |
| 行政法专家组 | 行政程序 | 行政、处罚、许可证、拆迁、行政复议、行政诉讼 |
| 经济法专家组 | 消费者权益 | 消费者、假货、退款、网购、公司、股东、食品安全 |
| 诉讼专家组 | 民事诉讼 | 起诉、法院、管辖、证据、保全、执行、时效 |

选组由 LLM 根据语义决定，关键词列表仅作路由参考，非硬规则。

#### 专家并行检索

被选中专家组内的细分专家**并行**执行 RAG 检索，每位专家：
1. 在本地 SQLite FTS5 索引中检索相关法条
2. 通过语义扩展词、同义词（`law_enhancements.db`）提升召回
3. 输出命中法条 + 简要分析意见

#### 协调员综合

各专家结果汇总后，协调员 LLM 整合为最终回答，严格限制只引用已检索到的法条。

---

### 第四步：公报案例检索与引用（所有路径均执行）

专家分析完成后，**无论走哪条路径**，都会执行公报案例检索，采用三路并行策略：

| 策略 | 方法 | 说明 |
|------|------|------|
| A - FTS | `searchGongbaoDocsMultiTerm` | 每个检索词独立 FTS5 全文搜索，绕过 2 字限制 |
| B - Keywords | `searchGongbaoByKeywords` | 对 `keywords` 字段做 LIKE 匹配，召回关键词精确命中的文书 |
| Note - 笔记匹配 | 字符集交集 | 对用户添加了笔记的文书做字符重叠判断（≥2 个汉字），优先推荐标注过的同类案例 |

**来源过滤**：`gongbao` 模式下 LLM 会解析问句意图，将 `sourceFilter` 设为 `"al"`（指导案例）、`"cpwsxd"`（裁判文书）或 `"sfwj"`（司法文件）；未明确指定时为 `nil`（全类型检索）。

**检索词来源**：coordinator 和 `gongbao` 模式均使用 LLM 扩词（口语化问句 → 2–6 个法律专业术语），作为 FTS 检索词。

**案例引用**：候选案例全部注入 coordinator 上下文，由 coordinator LLM 自主判断哪些值得引用。引用案例以 `【参考案例】《标题》：一句话说明` 格式写入回答末尾，下方自动展示对应的可点击卡片。未被 coordinator 引用的候选不展示。

## 数据库

应用内置两个 SQLite 数据库（Bundle 资源，只读）：

| 文件 | 大小 | 说明 |
|------|-----:|------|
| `law_content.db` | ~250MB | 主库：laws / nodes / nodes_fts / article_references / gongbao_* |
| `law_enhancements.db` | ~120KB | RAG 增强：term_aliases / topic_law_hints / keyword_synonyms |

主要表结构：

| 表 | 说明 |
|----|------|
| `laws` | 2,020 部法律元数据（主库 1,253 + 公报司法解释 767），含 `source`、`is_current`、`is_flk` 等字段 |
| `nodes` | ~84,000 条条文及编章节节点，含 `global_order`、`article_num` 等 |
| `nodes_fts` | FTS5 trigram 全文索引（外部内容表，≥3 字） |
| `nodes_fts_bigram` | FTS5 unicode61 索引（1–2 字短词） |
| `article_references` | 8,340 条法条间引用关系，解析率 96.2% |
| `gongbao_docs` | 公报文书 2,289 条（指导案例/司法文件/裁判文书） |
| `gongbao_case_law_links` | 公报文书 → 法条关联 3,533 条，解析率 99.8% |
| `gongbao_docs_fts` | 公报 FTS5 trigram 全文索引 |

数据库由 [laws-data](https://github.com/13098806890/laws-data) 项目的 pipeline 生成，每次发版随应用 Bundle 更新。

## 技术架构

| 组件 | 说明 |
|------|------|
| `DatabaseManager.swift` | SQLite 封装，单例，串行队列保证线程安全 |
| `LegalExpertService.swift` | 专家协作系统：意图分类 → 模式路由 → 专家选组 → 并行检索 → 综合协调 → 公报案例检索 |
| `ExpertGroups.swift` | 6 个专家组定义，含路由关键词与细分专家列表 |
| `LegalTypes.swift` | 核心类型：`QueryMode`、`MessageIntent`、`RAGEvent`、`GongbaoCitation` 等 |
| `LegalChatViewModel` | 对话状态管理，追问轮次控制，ViewModel 驱动 UI |
| `ChatHistoryStore.swift` | 对话历史持久化（iCloud Documents / 本地 Documents fallback） |
| `PurchaseManager.swift` | StoreKit 2 内购管理，免费次数 + 每周 Pro 配额，`ConsumedPath` 精确退还逻辑 |
| `GongbaoView.swift` | 公报浏览、搜索、案例笔记界面 |
| `LawDetailView.swift` | 法律全文阅读，条文锚点滚动，公报案例链接 |
| `UserStore.swift` | 用户偏好、收藏、公报笔记持久化（iCloud KV） |

LLM 接入：DeepSeek（`deepseek-chat` / `deepseek-reasoner`），API Key 存储于系统 Keychain。
